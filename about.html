About

This page contains information regarding different types of objects, methods and permissions utilized to make this application function.

Privacy

Chat lobby sessions are stateful. What this means is that after all users exit the lobby session or it is manually destroyed by the host, all information regarding that session is lost. Moreover, all information regarding to users and what sessions they are connected to is lost when a user disconnects from the server. This is all done in the interest of privacy. No data is retained because on the server end, users and lobbies only exist for the duration of their lifetime. For evidence of this, you may check the source, which is available on Github. If that is not satisfactory, please, by all means, set up your own mirror server! I would love to have multiple chat sites using this code. For more information, please check out this quick and dirty server setup guide.

Safety

By default, when you create a lobby, or when a person enters an active lobby, that lobby server side enters a "text only" mode. What this means is that when a lobby is first opened, the only permissable data that may be sent is text. The text is inseted as plaintext inside of a div container element. This allows people to communicate in a safe environment when new users are added into the lobby. To elevate the client to HTML mode, a vote process must take place, where all users cast their elevation request. The server will send a message to those users that "<Number_Of_Users_Voting> out of <Number_of_Users_In_Lobby> are voting to elevate permissions to: HTML." If all users have voted to elevate the permissions, the vote passes and the server side settings are changed.

Even if the server is elevated to HTML send permissions, the users still won't be able to recieve HTML unless they change their client recieve HTML permissions to on. This is on a client-by-client basis, so there is no voting involved. Once the user has their client settings set, their browser will accept and append all incoming messages as HTML.

There is also a Cross-Site-Script mode, where users can send one another functions and they will be appended to the other user's browser and run. This is a dangerous mode, and is not reccommended except for in cases where you know all of the parties in the chat lobby have no mal intent. This once again requires another vote to be passed so that users can be made aware that a person is voting to elevate permissions to allow for script dispersal, and make the decision on whether or not they would like to be echoed scripts from the server. Even with this setting enabled, they will still need to have their client settings changed to allow for cross site scripts to be appended to thier client HTML document.

At any time a user can disable their client settings or simply leave the lobby. Don't stay in a lobby where untrusted code is going to be present, you are only putting yourself at greater risk of being hacked.
